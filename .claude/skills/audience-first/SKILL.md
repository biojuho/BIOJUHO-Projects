## 1. 언제 이 스킬을 쓰는가

다음 중 하나라도 해당하면 청중 확인 단계를 먼저 진행한다:

| 요청 유형 | 트리거 예시 |
|-----------|------------|
| **코딩/앱** | "앱 만들어줘", "웹사이트", "대시보드", "서비스 개발" |
| **콘텐츠** | "블로그 글", "뉴스레터", "영상 스크립트", "소개글" |
| **디자인** | "랜딩페이지", "포스터", "UI 디자인", "슬라이드" |
| **기획** | "서비스 기획", "제품 로드맵", "출시 전략" |
| **배포/마케팅** | "배포", "런칭", "홍보", "포지셔닝" |
| **A/B 테스트** | "어떤 게 더 나아?", "두 버전 비교", "성과 측정" |

**예외** (청중 확인 생략해도 되는 경우):
- 명확히 개인 유틸리티 ("내 할 일 목록", "내가 쓸 스크립트")
- 이미 청중이 대화에서 충분히 정의된 경우
- 단순 코드 수정/버그 픽스

## 2. 청중 파악 프레임워크

### Step 1: 배포 범위 먼저 확인

```
이 산출물은:
  A) 외부 유저/고객에게 배포되는 것인가?  →  [외부용] 워크플로우
  B) 팀 내부에서만 쓰는 것인가?           →  [내부용] 워크플로우
  C) 콘텐츠 (글/영상/발표자료)인가?        →  [콘텐츠] 워크플로우
```

### Step 2: B2B vs B2C 구분 (v2.0 신규)

**외부용일 경우 추가 구분**:

| Type | 예시 | 핵심 차이 |
|------|------|----------|
| **B2C** | 뉴스레터, 트렌드 트윗, 개인용 앱 | 감성 > 논리, 빠른 의사결정, 바이럴 중요 |
| **B2B** | SaaS 대시보드, 엔터프라이즈 API | 논리 > 감성, 긴 구매 사이클, ROI 증명 필수 |
| **Prosumer** | 개발자 도구, 리서치 플랫폼 | 전문성 + 효율성, 문서 품질 중요 |

**결정 방법**:
- 구매자 ≠ 사용자? → B2B
- 단가 > $500 or 계약형? → B2B
- 개인이 직접 결정? → B2C
- 전문가용이지만 개인 구독? → Prosumer

### Step 3: 유형별 필수 확인 항목

각 워크플로우의 상세 내용은 `references/` 참조.
빠른 요약:

**[B2C 제품/콘텐츠]** → `references/b2c-product.md`
- 타깃 페르소나 (연령, 관심사, 소비 패턴)
- 감정적 니즈 (두려움, 욕망, 소속감)
- 바이럴 요소 (공유 동기)
- 첫 5초 훅 (이탈 방지)
- SNS 채널별 포맷 조정

**[B2B 제품/서비스]** → `references/b2b-product.md`
- 의사결정자 vs 실사용자 분리
- 조직 Pain Point (비용, 위험, 효율)
- ROI/TCO 계산 근거
- 경쟁사 대비 차별점
- 보안/컴플라이언스 요구사항
- 구매 사이클 (PoC → 계약 → 온보딩)

**[내부용 도구]** → `references/internal-tool.md`
- 사용자 (팀원 규모, 기술 수준)
- 현재 수동 프로세스와 Pain point
- 성공 지표 (시간 절약, 오류 감소 등)
- 유지보수 주체

**[콘텐츠 — 글/영상/발표]** → `references/content.md`
- 주요 독자/시청자 페르소나
- 독자의 현재 지식 수준
- 콘텐츠를 접하는 채널/맥락
- 독자가 콘텐츠 후 취해야 할 행동 (CTA)
- 원하는 감정적 반응

## 3. 실행 절차 (4 Phases)

### Phase 0: 청중 인터뷰 (항상 먼저)

요청을 받으면 **바로 만들지 말고**, 아래 질문 중 모르는 것을 먼저 물어본다.
이미 대화에서 답이 나온 항목은 건너뜀.
**한 번에 너무 많이 묻지 말 것** — 가장 중요한 2~3개만 먼저.

```
💡 시작 전 확인 체크리스트:

□ 누가 쓸 것인가? (타깃 유저/독자)
□ 그들의 가장 큰 문제/니즈는?
□ 외부 배포용인가, 내부용인가?
□ B2C/B2B/Prosumer 중 어디에 해당?
□ 이게 성공하면 어떤 모습인가? (성공 기준)
□ 비슷한 경쟁/대안이 있는가?
□ 어떤 언어/문화권인가? (i18n 고려)
```

### Phase 1: 청중 프로필 작성

인터뷰 후 산출물 시작 전에 간략한 청중 프로필을 먼저 제시:

```markdown
## 🎯 청중 프로필 (확인 후 진행)

**타입**: [B2C / B2B / Prosumer / Internal]
**타깃**: [페르소나 설명]
**핵심 니즈**: [그들이 진짜 원하는 것]
**배포 범위**: [외부/내부/콘텐츠]
**언어/문화**: [ko-KR / en-US / 글로벌]
**성공 기준**: [이게 잘 됐다는 기준]
**설계 방향**: [이를 바탕으로 어떻게 만들 것인지 한 줄 방향]
```

유저가 OK하면 그때 실제 산출물 제작 시작.

### Phase 2: 청중 중심 설계 반영

산출물 제작 시 청중 프로필을 계속 참고:

- **코드/UI**: UX 결정 시 타깃 기술 수준 반영 (비기술자 → 단순화, 개발자 → 파워유저 기능)
- **콘텐츠**: 어조(tone), 전문 용어 수준, 예시 선택 모두 독자 기준으로
- **구조**: 타깃이 원하는 정보를 먼저, 부가 정보는 뒤로
- **i18n**: 언어뿐 아니라 문화적 맥락 반영 (예: 한국 → 경제불안 민감도 높음)

### Phase 3: 마케팅 체크 (외부 배포용일 때)

외부용 제품/콘텐츠 완성 후 선택적으로 제공:

```
✅ 마케팅 셀프 체크
- 타깃이 처음 5초 안에 "이게 나를 위한 것"임을 알 수 있는가?
- 핵심 가치 제안(Value Proposition)이 명확한가?
- CTA(다음 행동 유도)가 있는가?
- 경쟁 대안 대비 차별점이 드러나는가?
- B2B라면: ROI가 명시되어 있는가?
- B2C라면: 감정적 공감을 유도하는가?
```

### Phase 4: 성과 측정 설계 (v2.0 신규)

**A/B 테스트 or 성과 추적이 필요한 경우**:

```
📊 성공 KPI 정의
- 정량 지표: [클릭률, 전환율, 체류시간, 공유수 등]
- 정성 지표: [사용자 피드백, 브랜드 인식 등]
- 측정 주기: [실시간 / 일일 / 주간]
- 의사결정 기준: [어떤 수치가 나오면 A/B 중 어느 쪽 선택?]
```

**A/B 테스트 체크리스트** (예: 두 콘텐츠 버전 비교):

```
□ 테스트 가설: "새 버전이 [지표]를 [%] 개선할 것"
□ 표본 크기: 최소 [N]명/회 이상
□ 테스트 기간: [시작일-종료일]
□ 측정 도구: [Google Analytics, Notion DB, 자체 트래킹]
□ 중단 조건: [심각한 부정적 반응 발생 시]
```

**예시: DailyNews Economy_KR A/B Test**

```python
# 기존 코드: 기술적 파이프라인만 비교
# 문제: "어느 쪽이 더 나은가?"를 사용자 반응으로 검증 안 함

# Audience-First 개선:
KPI = {
    "primary": "X 포스트 인게이지먼트율 (좋아요+리트윗+답글/조회수)",
    "secondary": ["평균 읽기 시간", "Notion 저장 수"],
    "target": "기존 대비 +15% 인게이지먼트",
    "sample_size": "각 버전 최소 20개 포스트",
    "decision_rule": "인게이지먼트 +10% 이상 → 새 버전 채택"
}
```

## 4. 산출물 유형별 빠른 가이드

### 코딩/앱 요청
1. 청중 프로필 확인
2. 기능 목록 작성 전에 "유저 스토리" 먼저:
   `"[타깃 유저]로서, [목표]를 달성하기 위해 [기능]이 필요하다"`
3. MVP 범위 제안 (처음부터 다 만들지 않기)
4. 외부 배포용이면 온보딩/빈 상태(Empty State) 설계 포함
5. **B2B라면**: 관리자/사용자 권한 분리 고려

### 블로그/뉴스레터/글
1. 독자 페르소나 + 읽는 맥락 확인
2. 제목 옵션 2~3개 먼저 제안 (클릭 유도)
3. 글 구조: 훅(Hook) → 문제 → 해결 → CTA
4. 전문 용어 수준 조정
5. **i18n**: 언어별 어조 조정 (영어 = 직접적, 한국어 = 완곡)

### 영상 스크립트
1. 시청자 페르소나 + 시청 플랫폼 확인
2. 첫 15초 훅 설계 (이탈 방지)
3. 시각 자료 요소 표시
4. 마지막 CTA 명시
5. **플랫폼 최적화**: YouTube (8분+), TikTok (60초), Instagram Reels (30초)

### 발표/슬라이드
1. 청중 배경 지식 수준 확인
2. 발표 후 청중이 할 행동/결정 명확히
3. 스토리라인 먼저, 슬라이드는 나중에

### A/B 테스트 (v2.0 신규)
1. **먼저 청중 가설 수립**: "어떤 사람이 A를 선호하고, 어떤 사람이 B를 선호할까?"
2. **측정 가능한 KPI 정의**: 주관적 "더 좋음" 금지, 정량 지표로
3. **표본 크기 계산**: 통계적 유의성 확보
4. **실험 설계**: 동일 시간대, 동일 채널, 랜덤 배정
5. **사후 분석**: 숫자만 아니라 "왜 A가 이겼는가?" 질적 분석

## 5. 예시 대화

**❌ 스킬 없이 (현재 문제)**
> 유저: "할 일 관리 앱 만들어줘"
> Claude: [바로 To-do 앱 코드 작성...]

**✅ 스킬 적용 후 (v2.0)**
> 유저: "할 일 관리 앱 만들어줘"
>
> Claude: "만들기 전에 방향을 잡으면 더 잘 만들 수 있어요.
>
> - 누가 주로 쓸 예정인가요? (혼자 쓸 개인 도구 vs 팀에 배포)
> - 기존에 쓰던 앱(Todoist, Notion 등)에서 불편했던 점이 있나요?
>
> **청중 프로필 (초안)**
> - 타입: B2C (개인용)
> - 타깃: 1인 창업자, 30대, 앱에 익숙함
> - 핵심 니즈: 빠른 입력 + 우선순위 관리
> - 배포 범위: 개인용 (내부)
> - 성공 기준: 매일 열어서 씀
> - 설계 방향: 입력 마찰 최소화, 복잡한 기능 배제
>
> 이 방향으로 만들까요?"

**❌ A/B 테스트 스킬 없이**
> 유저: "Economy_KR 콘텐츠 A와 B 중 어느 게 나아?"
> Claude: [주관적 의견: "A가 더 깔끔해 보이네요"]

**✅ A/B 테스트 스킬 적용 (v2.0)**
> Claude: "비교하기 전에 먼저 확인할게요:
>
> - Economy_KR 독자는 누구인가요? (투자자? 일반인?)
> - 어떤 지표로 '더 나음'을 판단하나요? (클릭률? 읽기 시간? 공유수?)
> - 실제 배포해서 측정 가능한가요, 아니면 예상만 하면 되나요?
>
> **추천 접근**:
> 1. 각 버전을 20회씩 X에 랜덤 게시
> 2. 인게이지먼트율(좋아요+RT/조회수) 측정
> 3. 통계적으로 유의미한 차이 나오면 승자 채택
>
> 이렇게 진행할까요, 아니면 바로 제 주관적 평가가 필요한가요?"

## 6. Localization/i18n 가이드 (v2.0 신규)

**언어별 청중 특성**:

| 언어/문화 | 어조 | 구조 | 감성 트리거 |
|----------|------|------|-------------|
| **한국어** | 완곡, 존댓말 기본 | 결론 뒤로 (서론-본론-결론) | 소속감, 경제불안, FOMO |
| **영어 (US)** | 직접적, 캐주얼 OK | 결론 먼저 (역피라미드) | 자기계발, 효율성, 독립성 |
| **일본어** | 극도로 완곡 | 맥락 중시, 암시적 | 조화, 집단주의 |
| **중국어** | 실용적, 수치 중시 | 증거 기반, 권위 인용 | 체면, 성공, 가족 |

**DailyNews 예시**: 한국 독자는 "부동산 위기"에 높은 감정 반응 → 어조 신중, 해결책 제시 필수

## 7. Persona Validation Checklist (v2.0 신규)

**청중 프로필을 작성했다면, 이걸로 검증하세요**:

```
□ 실제 데이터 기반인가? (아니면 추측?)
  → 있다면: Google Analytics, 설문조사, 인터뷰
  → 없다면: 가설임을 명시하고 검증 계획 수립

□ 구체적인가? ("2030 여성" X → "28세 스타트업 마케터, 노션 파워유저" O)

□ 행동 패턴이 포함되어 있나? (인구통계만 있으면 부족)
  → "언제, 어디서, 왜 이걸 쓰는가?"

□ Pain Point가 명확한가? (기능 아니라 문제 중심)

□ 경쟁 대안 사용 경험이 정리되어 있나?
  → "Notion은 느림, Todoist는 비쌈" 등

□ 성공/실패 시나리오가 구체적인가?
  → 성공: "매일 아침 5분 안에 일정 파악"
  → 실패: "설정이 복잡해서 1주일 후 포기"
```
