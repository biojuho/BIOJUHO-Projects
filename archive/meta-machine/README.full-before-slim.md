# JooPark Workspace v3.0

일정·할 일·메모·습관·통계를 한 화면에서 관리하는 **개인 워크스페이스**에, 프로젝트(PM)·DB 카탈로그 관리까지 더한 정적 SPA입니다. 빌드/패키지 없이 정적 파일만으로 동작하며, 내가 만든 모든 데이터는 브라우저(localStorage)에 자동 저장됩니다.

## 실행

```bash
python3 -m http.server 5178
```

브라우저에서 `http://127.0.0.1:5178/` 를 엽니다. (저장 기능은 `http://`에서 동작합니다. `file://`로 직접 열면 저장이 제한될 수 있습니다.)

서버를 켠 상태에서 아래 명령을 실행하면 설치된 Chrome headless/CDP로 16개 화면 라우트, 콘솔 오류, 네트워크 4xx/5xx, GitHub/도입 후보 데이터 병합 카운트를 한 번에 확인합니다.

## 출시 패키지 생성

공개 공유 이미지나 홈 화면이 바뀌었으면 먼저 실제 화면 기반 preview를 갱신합니다. 이 명령은 임시 정적 서버와 Headless Chrome을 사용해 `social-preview.png`를 1200×630 PNG로 다시 생성합니다.

```bash
npm run capture:preview
```

정적 호스팅에 올릴 파일만 `dist/release/`로 복사하고, 파일별 SHA-256 manifest와 실행 안내를 생성합니다.

```bash
node scripts/package-release.mjs
cd dist/release
python3 -m http.server 5178
```

`package-release`는 staging 디렉터리에서 전체 manifest와 `RELEASE.md`를 만든 뒤 `dist/release`를 교체하고, 작업 중에는 `dist/release.packaging.lock` release package lock을 둡니다. `verify-release`는 이 lock이 사라질 때까지 기다리므로 audit이나 verify를 병렬로 실행해도 반쯤 만들어진 `dist/release`를 release artifact로 읽지 않습니다. `autoresearch-results/release-readiness-summary.json` 또는 `autoresearch-results/verify-workspace-summary.json`이 깨끗한 CI checkout에 없으면 패키저는 `packageBootstrap`과 `validForExternalClaim: false`가 들어간 invalid/bootstrap JSON만 release artifact에 넣어 빌드 순환 의존을 끊습니다. 이 bootstrap은 런타임 수리 안내용이며, `npm run verify:full` 또는 `node scripts/audit-release-readiness.mjs --format=summary`로 실제 증거를 다시 만들기 전에는 외부 완료/공개 증거가 아닙니다.

`dist/release/`는 그대로 정적 호스팅에 업로드할 수 있습니다. 패키저는 GitHub Pages용 `404.html`, Netlify용 `_headers`/`_redirects`, Vercel용 `vercel.json`을 함께 생성하며, `node scripts/verify-release.mjs`가 이 배포 지원 파일들의 존재, 핵심 헤더/리다이렉트/CSP 설정, `dist/release`의 source-copied runtime file and seed asset source parity까지 검증합니다. 공개 링크 미리보기와 설치형 웹앱 표면은 `site.webmanifest`, Open Graph/Twitter 메타 태그, 실제 홈 화면 캡처인 `social-preview.png`, fallback/source preview인 `social-preview.svg`, `icons/icon-192.svg`, `icons/icon-512.svg`로 제공됩니다. `sw.js` service worker는 앱 shell, vendor 파일, 핵심 데이터 receipt를 offline app shell로 precache하고, 온라인에서는 network-first로 최신 파일을 먼저 받아 오래된 캐시 고착을 줄입니다. `app.js`는 secure context 또는 localhost에서만 service worker 등록을 시도하며, `verify-release`와 `smoke-release`는 `sw.js` 포함 여부, registration term, offline shell cache term, `Cache-Control: no-cache` 헤더를 함께 검사합니다.

Pages workflow attestation readiness는 `docs/github-pages-workflow.yml`, `.github/workflows/joopark-pages.yml`, `prepare-github-pages-workflow.mjs`, `plan-workflow-ui-install.mjs`, `plan-publish-dispatch.mjs`, `install-remote-workflow-files.mjs`가 모두 `attestations: write`, `actions/attest@v4`, `subject-path: dist/release/**`를 required terms로 공유할 때만 인정합니다. 이 단계는 `node scripts/verify-release.mjs` 뒤, `actions/upload-pages-artifact@v4` 앞에서 release package subject를 attestation 처리하도록 설계되어 있으며, 원격 workflow 설치와 Actions 실행 뒤에야 서명된 GitHub artifact attestation proof로 볼 수 있습니다.

System Status의 `Pages attestation proof intake`는 원격 Pages workflow가 실행된 뒤 `actions/attest@v4` 출력과 검증 결과를 붙여 넣기 위한 copy-ready 증거 수집 패킷입니다. 이 intake는 `attestation-url`, `attestation-id`, `gh attestation verify dist/release/release-manifest.json`, `gh attestation verify dist/release/index.html`, predicate type, Pages workflow run URL/headSha를 같은 receipt에 모읍니다. 현재 상태는 `not signed proof yet`이며, receipt guard는 `Do not claim signed GitHub artifact attestation proof`를 유지합니다. 원격 workflow가 성공하고 두 `gh attestation verify` 명령이 통과하며 `capture-publish-evidence`와 `verify-launch-handoff`가 최종 공개 proof gate를 ready로 보고하기 전까지 외부 공개 완료, archive proof, `readyForExternalClaim=true`를 주장하지 않습니다.

체크섬 manifest와 source parity까지 함께 검증하려면 먼저 `node scripts/verify-release.mjs`를 실행합니다. 패키징은 `release-manifest.json` 외에 `release-provenance.json`도 생성합니다. 이 파일은 in-toto Statement v1 형식과 SLSA provenance v1 predicate를 따르는 unsigned-local-provenance 증거이며, `release-manifest.json` SHA-256을 subject digest로 기록하고 `source-tree`, `index.html`, `app.js`, `sw.js`, `data`, `vendor` resolvedDependencies를 남깁니다. `scripts/verify-release.mjs`는 이 subject digest, builder id, source commit/branch/dirty state, runtime file count, total bytes, `signed=false` 상태를 함께 확인합니다. System Status의 `Release provenance` 패널은 같은 `release-provenance.json`을 읽어 subject digest, predicate/build type, builder id, source commit, dependency count, `unsigned-local-provenance`, `node scripts/verify-release.mjs`, `npm run verify:full`을 보여주고 `provenance receipt` 복사를 제공합니다. `releaseProvenanceUiCoverageScore`는 이 loader, System Status panel, smoke coverage, README handoff가 모두 준비됐을 때만 4/4로 인정합니다. 이 로컬 provenance는 GitHub artifact attestations처럼 workflow에서 서명된 attestation이 아니므로, remote workflow 설치와 Actions 실행 전에는 외부 공개 완료 증거로 주장하지 않습니다.

System Status의 `Verify workspace summary` 패널은 릴리스 패키지와 service worker precache에 포함되는 `autoresearch-results/verify-workspace-summary.json`을 읽어 `release_readiness_gates`, `launch_readiness_refresh`, `product_loop_summary_sync`, `evidenceSync`, `safeToDispatch`, `readyForExternalClaim`, step duration, `npm run verify:full` command를 보여주고 `JooPark Verify Workspace Summary Receipt`를 복사합니다. `npm run refresh:launch-readiness`와 `node scripts/sync-product-loop-summary.mjs --write --markdown`은 여전히 개별 복구 명령으로 사용할 수 있습니다. 기본 verify와 full verify를 분리해 파일 쓰기 감시 환경에서 refresh artifact가 다시 `npm run verify`를 호출하는 루프를 만들지 않으면서도, 운영자는 `npm run verify:full` 하나로 release gate와 evidence sync를 함께 검증할 수 있습니다. audit 실행은 compact summary cache도 `autoresearch-results/release-readiness-summary.json`에 저장하며, 이 파일은 `capture-output-quality-audit`이 재귀 audit 없이 최신 pass/fail count를 읽는 non-recursive latest gate source입니다. 이후 `node scripts/audit-release-readiness.mjs --format=summary`는 source commit, source file SHA-256 fingerprint, 6시간 freshness window가 그대로 맞을 때만 저장된 packaged browser gate를 재사용하며, 맞지 않으면 다시 `--run-gates`를 요구합니다. 이 fingerprint는 파일 수정 시간 대신 내용을 해시하므로 같은 내용의 증거 JSON이 다시 쓰여도 캐시가 즉시 stale 처리되지 않습니다. 이 fingerprint는 감사 실행 끝에 다시 쓰이는 `data/launch-execution-packet.json`, `data/launch-readiness-refresh.json`, `data/launch-readiness-refresh.md`, `data/output-quality-audit.json`, `data/publish-evidence.json` 자기 산출물, release manifest와 정적 audit에서 별도로 검증되는 `README.md`, release artifact가 아닌 AutoResearch 운영 summary인 `autoresearch-results/joopark-product-loop.json`과 `autoresearch-results/joopark-product-loop.md`, full verify runner가 다시 쓰는 `autoresearch-results/verify-workspace-summary.json`, 자기 검증 때문에 자주 바뀌는 `scripts/audit-release-readiness.mjs`와 output-quality receipt generator인 `scripts/capture-output-quality-audit.mjs`를 제외해 방금 만든 fresh packaged browser gate가 즉시 stale 처리되지 않게 합니다. `--run-gates`의 release readiness audit lock은 `owner.json`의 PID를 `process.kill(pid, 0)`로 확인하고 실제 명령이 `scripts/audit-release-readiness.mjs --run-gates`인지 비교하므로, 죽은 owner는 `owner_process_missing`, PID 재사용은 `owner_pid_reused` stale release readiness audit lock으로 즉시 output-quality receipt, publish evidence, dispatch plan, workflow UI install plan, remote workflow file check, launch execution packet, launch handoff verification, main bridge plan을 나열하므로 `--product-loop`, `--release-gate-cache`, `--release-readiness-summary`, `--previous-output-quality`, `--launch-handoff-verification`, `--main-bridge-plan`을 바꾼 fixture 실행도 실제 입력 출처를 숨기지 않습니다. 전체 checklist가 필요하면 `node scripts/audit-release-readiness.mjs --run-gates --format=markdown`, compact machine-readable JSON 증거가 필요하면 `node scripts/audit-release-readiness.mjs --run-gates`를 사용합니다. 기본 `--format=json`은 Node `spawnSync` 기본 버퍼에서도 소비되도록 compact machine-readable JSON을 출력하고, 사람이 읽는 formatted JSON은 `--format=json-pretty` 또는 `--pretty`로 파일 리다이렉트해 확인합니다.

## AI prompt handoff

포트폴리오의 `벤치 포커스` 화면은 Workspace, Knowledge/IA, PM benchmark 후보를 비교한 뒤 복사 가능한 Markdown handoff를 제공합니다. 이 handoff는 단순 설명문이 아니라 `joopark-review-handoff/v2` 계약을 따르는 실행용 프롬프트 패키지입니다.

- `System Prompt`는 역할, 권한 우선순위, 후보 설명을 지시가 아닌 데이터로 취급하는 규칙을 고정합니다.

## LLM 위키 운영

`LLM 위키`는 로컬 워크스페이스 안에서 LLM 기초, 프롬프트, 모델 선택, Claude API, MCP/RAG/에이전트, 평가·안전·비용을 한국어로 빠르게 확인하는 오프라인 지식 베이스입니다. 콘텐츠는 `llm-wiki-view.js`의 `WIKI` 데이터에 임베드되어 있고, 항목별 `sources`는 같은 파일의 `WIKI_SOURCES` 공식 출처 레지스트리를 참조합니다.

- 출처 원칙: 공식 문서·표준·논문을 우선하고, 모델명·가격·컨텍스트·API 파라미터처럼 변동이 큰 주장은 30일이 지나면 stale 후보로 봅니다. 재확인 전 비교 수치는 `불확실`로 표시합니다.
- 구조 선택: 이번 버전은 no-build 정적 SPA와 source parity 검증을 유지하기 위해 코드 임베드 + 구조화 출처 메타데이터를 채택했습니다. 별도 JSON/Markdown 소스 분리는 loader/cache/source parity를 함께 설계하는 다음 구조화 사이클 후보입니다.
- 모델 지형 갱신: `landscape`, `context-window`, `choosing`은 `openai_models`, `openai_api_platform`, `google_gemini_models`, `google_gemini_pricing`, `meta_llama4`, `deepseek_pricing`를 함께 확인합니다. `GPT-5.5`, `Gemini 3.1 Pro Preview`, `DeepSeek V4 Flash`, `Llama 4 Scout` 같은 가격·컨텍스트 행은 task-tiered model matrix 기준으로 비교하고, tiered/cache/self-hosted 가격을 같은 단가처럼 섞지 않습니다.

### candidate freshness drift

```bash
node scripts/check-candidate-freshness-drift.mjs --snapshot-only
```

실제 GitHub 현재값과 비교하려면 `gh` 인증이 있는 환경에서 live 모드를 사용합니다. `--live`는 drift를 JSON으로 보고하고, 자동화에서 drift를 실패로 처리하려면 `--fail-on-drift`를 함께 붙입니다. `lastCommit`, `pushedAt`, issue/PR, disk drift는 기본적으로 `blockingDriftCount`와 `actionableDriftCount`에 포함되어 실패 조건이지만, 고빈도 popularity 값인 `stars/forks`는 `advisoryDriftCount`로만 보고합니다. `lastCommit`이 그대로인 `pushedAt` 또는 `diskKb` drift는 default-branch source freshness 변화가 아니므로 `metadata-advisory`로 보고합니다.
GitHub REST `open_issues_count`는 PR을 포함하지만 GraphQL은 issue와 PR을 분리하므로, monitor는 `openIssues` snapshot을 issue-only 또는 issue+PR total 중 하나와 일치하면 유효한 값으로 봅니다.

```bash
node scripts/check-candidate-freshness-drift.mjs --live
node scripts/check-candidate-freshness-drift.mjs --live --fail-on-drift
```

반복 실행은 `docs/github-drift-watch-workflow.yml` 템플릿을 repository root의 `.github/workflows/joopark-drift-watch.yml`로 복사해 **Watch JooPark Candidate Drift** workflow로 운영합니다. 이 workflow는 매주 월요일 `schedule`로 advisory live drift를 보고하고, `workflow_dispatch`에서 `mode=fail-on-drift`를 선택하면 blocking drift를 실패로 처리합니다. `repo` 입력으로 `owner/name` 하나를 좁혀 볼 수 있습니다. GitHub workflow syntax 기준으로 scheduled workflow와 manual dispatch는 default branch의 workflow 파일에서 동작하므로, 먼저 템플릿을 기본 브랜치에 설치해야 합니다. live 단계는 GitHub CLI가 요구하는 `GH_TOKEN`에 `${{ secrets.GITHUB_TOKEN }}`를 전달하고, 권한은 `contents: read`만 사용합니다.

```bash
node scripts/prepare-github-drift-watch-workflow.mjs --dry-run
node scripts/prepare-github-drift-watch-workflow.mjs --dry-run --check-scope
node scripts/prepare-github-drift-watch-workflow.mjs --write
```

`--dry-run --check-scope`는 현재 GitHub 토큰의 `workflowScopeAvailable` 값을 JSON으로 보고합니다. `.github/workflows/joopark-drift-watch.yml` 설치는 `workflow` scope가 있는 토큰이나 GitHub UI 세션에서만 `--write`로 수행합니다. 현재 Pages publish workflow와 같은 GitHub token 정책을 따르므로, CLI 토큰에 `workflow` scope가 없으면 템플릿 검증까지만 통과하고 실제 repository-root workflow 파일은 쓰지 않습니다.

빠르게 움직이는 후보만 별도로 확인하려면 `--repo owner/name`을 붙입니다. 이 필터는 반복할 수 있고 `--snapshot-only`와 `--live` 모두에서 동작합니다.

```bash
node scripts/check-candidate-freshness-drift.mjs --live --repo Veritas-7/autoresearch-skill-system
```

high-churn 후보인 `Veritas-7/autoresearch-skill-system`은 출시 게이트 전, upstream 이동 감지 후, `--fail-on-drift` 자동화를 켜기 전에 repo-scoped cadence로 먼저 확인합니다. 이 후보는 issue/PR 메타데이터가 그대로이고 drift가 `lastCommit`, `pushedAt`, `diskKb`에만 한정되며 live `pushedAt`이 snapshot `pushedAt`보다 4시간 이내일 때 `cadence-advisory`로 보고되어 release deadlock을 만들지 않습니다. 정책 증거는 네트워크 없는 snapshot-only 모드에서 확인할 수 있습니다.

```bash
node scripts/check-candidate-freshness-drift.mjs --snapshot-only --repo Veritas-7/autoresearch-skill-system --cadence-policy
node scripts/check-candidate-freshness-drift.mjs --live --repo Veritas-7/autoresearch-skill-system
node scripts/check-candidate-freshness-drift.mjs --live --repo Veritas-7/autoresearch-skill-system --fail-on-drift
```

반복되는 `Veritas-7/autoresearch-skill-system` 스냅샷 갱신은 writer로 먼저 dry-run한 뒤, 결과를 확인하고 `--write`로만 파일을 수정합니다. 출시 감사는 네트워크 없는 `--snapshot-only` 모드로 현재 로컬 row shape와 source marker를 확인합니다.

```bash
node scripts/refresh-veritas-candidate-snapshot.mjs --snapshot-only
node scripts/refresh-veritas-candidate-snapshot.mjs --dry-run
node scripts/refresh-veritas-candidate-snapshot.mjs --dry-run --fail-on-change
node scripts/refresh-veritas-candidate-snapshot.mjs --write
```

다른 source-backed 도입 후보는 generic writer로 같은 절차를 사용합니다. `--repo owner/name`으로 대상 row를 지정하고, dry-run으로 변경 범위를 본 뒤 `--write`로만 `data/adoption-candidates.json`을 갱신합니다.

```bash
node scripts/refresh-candidate-snapshot.mjs --snapshot-only --repo Veritas-7/autoresearch-skill-system
node scripts/refresh-candidate-snapshot.mjs --dry-run --repo outline/outline
node scripts/refresh-candidate-snapshot.mjs --dry-run --repo outline/outline --fail-on-change
node scripts/refresh-candidate-snapshot.mjs --write --repo outline/outline
node scripts/refresh-candidate-snapshot.mjs --dry-run --from-live-drift
node scripts/refresh-candidate-snapshot.mjs --write --from-live-drift
node scripts/refresh-candidate-snapshot.mjs --dry-run --from-live-drift --actionable-only
```

현재 live drift가 있는 row만 한 번에 갱신할 때는 `--from-live-drift`를 사용합니다. 이 모드는 먼저 `check-candidate-freshness-drift.mjs --live`로 drifted repo 목록을 만들고, 그 목록에 해당하는 source-backed rows만 refresh합니다. release gate를 실제로 막는 row만 갱신하려면 `--actionable-only`를 함께 붙여 `actionableDrifted` 후보로 좁힙니다. 특정 repo 집합으로 좁히려면 `--repo owner/name` 필터를 함께 반복해서 붙입니다.

- `data/repos.json` — GitHub 저장소 스냅샷 (포트폴리오 시드) · `data/adoption-candidates.json` — OSS 도입 후보 스냅샷 · `scripts/sync-github.sh` — GitHub 스냅샷 재생성
- `vendor/` — 로컬 동봉한 오픈소스 라이브러리 (아래 "도입한 오픈소스" 참고) · `vendor/LICENSES.md`

## 도입한 오픈소스

빌드 도구 없이 동작하도록, 검증된 경량·무의존 OSS를 단일 파일(UMD)로 `vendor/`에 **로컬 동봉**했습니다. CDN 런타임에 의존하지 않아 오프라인(`file://`)에서도 그대로 동작합니다.

| 라이브러리 | 버전 | 라이선스 | 용도 |
| --- | --- | --- | --- |
| [Fuse.js](https://github.com/krisk/Fuse) | 6.6.2 | Apache-2.0 | 명령 팔레트 퍼지(오타 허용)·관련도 랭킹 검색 |
| [marked](https://github.com/markedjs/marked) | 18.0.5 | MIT | 메모 본문 Markdown 렌더 |
| [DOMPurify](https://github.com/cure53/DOMPurify) | 3.4.8 | Apache-2.0 / MPL-2.0 | marked 출력 HTML XSS 소독 |

라이브러리가 로드되지 않은 환경에서도 앱은 정상 동작합니다(검색은 부분일치, 메모는 평문으로 자동 폴백). 여전히 **외부 빌드/패키지 매니저 없이 정적 파일만으로** 동작합니다.

## GitHub 동기화 (선택)

```bash
gh auth status          # 로그인 확인
./scripts/sync-github.sh
```

`gh` 인증이 만료되었거나 없으면 공개 GitHub API로 보이는 저장소를 갱신하고, 공개 API에서 보이지 않는 기존 로컬 스냅샷 항목은 보존합니다.

부팅 시 `data/repos.json`과 `data/adoption-candidates.json`을 읽어 포트폴리오를 시드합니다. 편집·저장한 프로젝트 데이터는 덮어쓰지 않으며, 도입 후보는 누락된 항목만 1회 병합합니다. 파일이 없거나 `file://`로 열면 내장 mock으로 동작합니다.

## Workflow UI Install Receipt

`node scripts/plan-workflow-ui-install.mjs --dry-run --write`는 `data/workflow-ui-install-plan.json`에 `JooPark GitHub UI Workflow Install Receipt`와 `JooPark GitHub UI Workflow Paste Packet`을 함께 저장합니다. System Status의 `UI install receipt 복사` / `UI paste packet 복사`는 GitHub UI new-file page에서 두 workflow를 만든 뒤 `check-remote-workflow-files.mjs --repo biojuho/BIOJUHO-Projects --write`, `gh workflow list --repo biojuho/BIOJUHO-Projects --all --json name,path,state,id`, `plan-publish-dispatch.mjs --live --repo biojuho/BIOJUHO-Projects --write`, `verify-launch-handoff.mjs --repo biojuho/BIOJUHO-Projects --write --markdown`를 순서대로 실행하도록 복사합니다. `workflowUiInstallPastePacketCoverage`는 이 paste packet이 렌더링/복사 가능한지 확인하는 smoke evidence입니다.

이 receipt는 설치 완료 증명이 아닙니다. `every post-install evidence field has been filled`, `remoteWorkflowFilesReady=true`, `remoteWorkflowVisibilityReady=true`, `dispatchReady=true`, `driftDispatchReady=true`, `allDispatchReady=true`, `verify-launch-handoff reports safeToDispatch=true`가 모두 확인되기 전까지 `gh workflow run`을 실행하지 않습니다.
