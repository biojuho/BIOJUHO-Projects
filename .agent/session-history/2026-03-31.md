### 진행 상황 요약

사용자가 `openclaw/acpx`의 `examples/flows/pr-triage` 스타일을 선호한다고 공유했고,
해당 패턴이 이 저장소에 실제로 필요한지 먼저 검토한 뒤, 과한 런타임 의존성은 빼고
우리 모노레포에 맞는 경량 triage 시스템으로 적응 도입했다.

### 판단

**채택한 것**:
- intention-first PR 작성/리뷰 원칙
- 변경 경로 기반 영향 영역 triage
- 사람 판단이 필요한 PR을 조기에 표면화하는 규칙
- 유지보수성 변경에는 과도한 bespoke 검증 강요를 줄이는 방향

**채택하지 않은 것**:
- persistent ACP session
- autonomous close / land
- 로컬/원격 상태를 강하게 재조정하는 stateful PR runtime

### 변경 파일

| 파일 | 유형 | 이유 |
|------|------|------|
| `.github/pull_request_template.md` | 수정 | PR 작성 단계에서 intent/problem/validation 강제 |
| `.github/workflows/pr-triage.yml` | 생성 | PR triage summary + artifact + sticky comment |
| `ops/scripts/pr_triage.py` | 생성 | deterministic triage 계산기 |
| `docs/PR_TRIAGE_SYSTEM.md` | 생성 | 왜 full ACPX 대신 adapted lane인지 설명 |
| `tests/test_pr_triage.py` | 생성 | triage 규칙 회귀 방지 |
| `TASKS.md` | 수정 | 완료 작업 기록 |
| `HANDOFF.md` | 수정 | 다음 작업자를 위한 운영 기록 |

### 검증 결과

- `python -m pytest tests/test_pr_triage.py -q` -> `9 passed`
- `python -m py_compile ops/scripts/pr_triage.py` -> exit `0`
- 로컬 smoke:
  - `python ops/scripts/pr_triage.py --base HEAD~1 --head HEAD --title "smoke: triage" --output-dir var/pr-triage-smoke`

### 핵심 교훈

- ACPX의 핵심 가치는 런타임이 아니라 **의도를 먼저 읽는 리뷰 습관**에 있다.
- 이 저장소는 이미 프로젝트별 GitHub Actions가 많아서, PR close/land 권한까지 가진 자율 런타임은 아직 과하다.
- 먼저 필요한 것은 merge automation이 아니라 **좋은 PR 입력과 명시적인 사람 판단 신호**다.
