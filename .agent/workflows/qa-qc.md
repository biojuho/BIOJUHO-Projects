# 🔧 Vibe Coding QA/QC 워크플로우 (v4.0 - GStack Enhanced)

> 사용법: `/qa-qc` 또는 코드 작성 요청 시 "QA/QC 포함" 키워드로 트리거
> 워크플로우 순서: STEP 1 → STEP 2 → STEP 3 → STEP 4
> **마스터 모드**: 4단계를 한 번에 순차 실행

---

## 모드 감지

사용자가 모드를 지정하지 않으면 **마스터 모드** (4단계 일괄 실행)로 동작한다.
- `step1`, `개발만` → STEP 1만 실행
- `step2`, `QA만` → STEP 2만 실행 (이전 코드 필요)
- `step3`, `수정만` → STEP 3만 실행 (이전 코드 + QA 결과 필요)
- `step4`, `QC만` → STEP 4만 실행 (최종 코드 필요)
- 기본값 / `전체` / `마스터` → STEP 1~4 순차 실행

## 자동 적용/스킵 판단 매트릭스

QA/QC 실행 전 변경 유형을 판별하여 자동으로 적용 수준을 결정한다:

| 변경 유형 | QA/QC 수준 | 설명 |
|-----------|:----------:|------|
| `.py`/`.js`/`.ts`/`.jsx`/`.tsx` 로직 변경 | ✅ 풀 (4단계) | 기본 워크플로 전체 실행 |
| 테스트 코드만 변경 | ⚡ STEP 2만 | QA 검토로 테스트 품질 확인 |
| `.md`/`.yml`/`.json` 설정 변경 | ⏭️ 스킵 | 비기능적 변경 |
| 주석/포맷팅만 변경 | ⏭️ 스킵 | 비기능적 변경 |
| 보안 관련 변경 (`.env`, auth, crypto) | 🔴 강화 모드 | 보안축 가중치 2배 + 추가 체크 |
| 인프라/CI 변경 (Dockerfile, workflow) | ⚡ STEP 2만 | 구성 검토만 |

## ✅ STEP 1 — 개발 (시니어 엔지니어 역할)

시니어 소프트웨어 엔지니어로서 사용자의 작업 요청을 구현한다.

### 개발 규칙
- 가독성 높은 코드를 작성하고 함수/변수명은 명확하게 지을 것
- 모든 함수에는 간단한 주석을 포함할 것
- 환경변수, API 키, 시크릿은 절대 하드코딩하지 말 것
- 엣지 케이스(빈 값, null, 예외 상황)를 반드시 고려할 것
- 코드 하단에 유닛 테스트 코드를 함께 작성할 것

### 출력 형식
1. 구현 코드
2. 테스트 코드
3. 사용 방법 (3줄 요약)

---

## ✅ STEP 2 — QA 자동 검토 + 자동 수정 (QA 엔지니어 역할)

날카로운 QA 엔지니어로서 STEP 1의 코드를 감정 없이 검토한다.

### 검토 항목 (5가지 축 — 반드시 전부 확인)

#### [기능성]
- 요구사항을 완전히 충족하는가?
- 누락된 기능이 있는가?
- 입력값 유효성 검사(Validation)가 있는가?

#### [보안 (Security)]
- `gitleaks` 스캔을 통과하는가? (`gitleaks detect -v` 실행 권장)
- SQL Injection, XSS, CSRF 취약점이 없는가?
- 민감한 정보(키, 비밀번호)가 코드에 노출되지 않는가?
- 외부 입력값을 신뢰 없이 처리하는 부분이 없는가?

#### [안정성]
- 예외 처리(try-catch, error handling)가 충분한가?
- 무한 루프, 메모리 누수 가능성이 없는가?
- 비동기 처리 오류 가능성이 없는가?

#### [코드 품질]
- 중복 코드(DRY 원칙 위반)가 없는가?
- 함수가 단일 책임 원칙(SRP)을 지키는가?
- 테스트 코드가 실제로 핵심 로직을 검증하는가?

#### [성능]
- O(n²) 이상의 불필요한 알고리즘 복잡도가 없는가?
- 불필요한 API 호출 반복, DB 쿼리 N+1 문제가 없는가?
- 비동기 처리에서 `await` 누락, 동기 블로킹이 없는가?
- 대용량 데이터 처리 시 메모리 효율적인가? (generator, streaming 등)

### 🔧 자동 수정 (AUTO-FIX) 패턴 ← v4.0 신규

> **영감**: [garrytan/gstack](https://github.com/garrytan/gstack)의 `/review` Fix-First 패턴

**명백한 기계적 문제는 즉시 자동 수정**하고 보고만 한다:
- 데드 코드 제거
- N+1 쿼리 최적화
- stale 주석/문서 업데이트
- 미사용 import 정리
- 타입 힌트 누락 보완

**모호한 이슈는 사용자 판단 요청** (수정하지 않고 플래그만):
- 보안 관련 결정
- 레이스 컨디션 / 동시성 설계
- API 계약 변경
- 비즈니스 로직 해석 차이

### 📐 완성도 갭 (Completeness Gap) 플래그 ← v4.0 신규

80% 구현에서 100% 구현까지의 추가 비용이 **30분 이내**면 플래그한다:
- "이 함수에 에러 핸들링을 추가하면 ~15분으로 완성도 100%"
- "이 API에 rate limiting을 추가하면 ~20분으로 프로덕션 레디"
- 30분 초과 예상이면 TODO로 등록하고 넘어감

### 출력 형식

```
QA 결과: [PASS ✅ / FAIL ❌]

### 자동 수정 완료 (AUTO-FIXED)
| 번호 | 파일:라인 | 문제 | 수정 내용 |
|------|----------|------|-----------|
| 1    |          |      |           |

### 사용자 판단 필요 (ASK)
| 번호 | 심각도 | 문제 내용 | 수정 제안 |
|------|--------|----------|-----------|
| 1    |        |          |           |

### 완성도 갭 (COMPLETENESS GAP)
| 번호 | 현재 상태 | 100% 달성 방법 | 예상 시간 |
|------|----------|---------------|-----------|
| 1    |          |               |           |

### 발견된 문제 목록
| 번호 | 심각도 (CRITICAL/HIGH/MED/LOW) | 카테고리 | 문제 내용 | 수정 방향 |
|------|-------------------------------|---------|-----------|-----------|
| 1    |                               |         |           |           |

### 총평
(문제 없으면 "이상 없음. STEP 3으로 진행하세요." 출력)
```

**심각도 기준:**
- **CRITICAL**: 보안 취약점, 데이터 손실 위험 → 즉시 수정 필수
- **HIGH**: 기능 오류, `await` 누락, 무한 루프 → 배포 전 수정
- **MED**: 코드 품질, 성능 개선 → 권장
- **LOW**: 스타일, 네이밍 → 선택

- **PASS**: AUTO-FIX만 있고 ASK 항목이 없으면 STEP 3를 스킵하고 바로 STEP 4로 진행
- **FAIL**: ASK 항목 또는 문제 목록과 함께 STEP 3로 진행

## ✅ STEP 3 — 코드 수정 + 회귀 테스트 생성 (시니어 엔지니어 역할)

QA 검토 결과를 반영하여 코드를 개선한다.

### 수정 규칙
- QA에서 지적된 **모든** 항목을 빠짐없이 수정할 것
- 수정하지 않은 항목이 있다면 반드시 이유를 명시할 것
- 수정으로 인해 기존 기능이 깨지지 않도록 할 것
- 수정된 부분에는 주석으로 `# [QA 수정]` 태그를 달 것

### 🧪 회귀 테스트 자동 생성 ← v4.0 신규

> **영감**: [garrytan/gstack](https://github.com/garrytan/gstack)의 `/qa` 자동 회귀 테스트 패턴

**수정된 모든 버그에 대해 회귀 테스트를 자동 생성**한다:
- 버그의 원래 증상을 재현하는 테스트 → 수정 전에는 실패, 수정 후 통과 확인
- 테스트 이름: `test_regression_<버그설명>_<날짜>`
- 엣지 케이스 테스트도 함께 추가 (동일 패턴의 다른 입력)

```
### 생성된 회귀 테스트
| 번호 | 테스트명 | 대상 버그 | 검증 내용 |
|------|---------|---------- |-----------|
| 1    |         |           |           |
```

### 출력 형식
1. 수정 완료된 최종 코드 (전체)
2. 수정 사항 요약 (항목별)
3. 생성된 회귀 테스트 목록
4. 수정하지 못한 항목 및 이유 (있을 경우)

---

## ✅ STEP 4 — QC 최종 승인 보고서 (QC 매니저 역할)

최종 배포 전 공식 QC 보고서를 작성한다.

### QC 보고서 작성 항목

#### 1. 요구사항 충족 여부
- 원래 요청한 기능이 모두 구현되었는가? (Yes/No + 근거)

#### 2. 최종 체크리스트
- [ ] 코드가 실행 가능한 상태인가?
- [ ] 유닛 테스트를 통과하는가? (로컬 테스트 단위 실행 결과 포함)
- [ ] 보안 취약점이 제거되었는가? (`gitleaks` 결과 패스)
- [ ] 환경변수 처리가 올바른가?
- [ ] 예외 처리가 완료되었는가?
- [ ] 주석 및 문서화가 충분한가?
- [ ] 성능 이슈가 해결되었는가?

#### 3. 예상 리스크
배포 후 발생 가능한 문제점을 LOW/MED/HIGH/CRITICAL로 분류

#### 4. 테스트 방법
운영 반영 전 확인해야 할 테스트 시나리오 3가지

#### 5. 롤백 계획
문제 발생 시 원복 방법

#### 6. 최종 판정

```
배포 승인: [✅ 승인 / ⚠️ 조건부 승인 / ❌ 반려]
판정 이유:

---
보고서 작성일: (오늘 날짜)
검토 버전: v1.0
```

### QA 결과 아카이빙

중요한 QC 보고서는 다음 경로에 저장한다:
```
d:\AI 프로젝트\.agent\qa-reports\YYYY-MM-DD-<subject>.md
```
예: `.agent/qa-reports/2026-03-08-healthcheck-v2.md`

> 아카이빙 대상: CRITICAL/HIGH 이슈가 1개 이상 발견된 경우, 또는 사용자 요청 시.
