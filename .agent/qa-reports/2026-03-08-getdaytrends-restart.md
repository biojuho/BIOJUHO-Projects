## 2. QA 검토 결과 (5축)

### [기능성] ✅ PASS
- `run_scheduler.bat`이 중복 실행 방지 로직 포함 (tasklist 기반)
- 스케줄러 모드 (`one_shot=False`) 정상 작동 확인
- 파이프라인 정상 완료: `collected=10, scored=10, errors=0`

### [보안] ✅ PASS
- `run_scheduler.bat`에 하드코딩된 시크릿 없음
- 스케줄 작업은 현재 사용자 컨텍스트에서 실행 (HIGHEST 권한)

### [안정성] ⚠️ 조건부 PASS
- 중복 실행 방지: `WINDOWTITLE` 기반 필터는 제목이 일치할 때만 작동
- 리스크: cmd 창 제목이 변경되면 중복 실행 가능 (LOW)
- 프로세스가 비정상 종료 시 자동 재시작은 다음 로그온까지 대기 필요

### [코드 품질] ✅ PASS
- bat 파일이 간결하고 목적이 명확
- 로그 기록 포함 (`run_scheduler.log`)

### [성능] ✅ PASS  
- 파이프라인 실행 시간: 약 122초 (정상 범위)
- 비용: $0.53/실행 (예산 범위 내)

## 4. 예상 리스크

| 심각도 | 리스크 | 대응 방안 |
|--------|--------|-----------|
| LOW | cmd 제목 변경 시 중복 실행 가능 | PID 파일 기반 점검으로 업그레이드 가능 |
| LOW | PC 재부팅 없이 프로세스 죽으면 다음 로그온까지 대기 | 2시간마다 헬스체크하는 추가 스케줄 작업 등록 권장 |
| MED | SIGINT 원인 불명 (외부 요인) | 시그널 원인 추적을 위한 상세 로깅 추가 권장 |

## 5. 테스트 시나리오

1. **정상 작동 확인**: 로그 파일에서 `pipeline_metrics` 라인이 2시간마다 기록되는지 확인
2. **재부팅 테스트**: PC 재시작 후 `GetDayTrends_AutoStart` 스케줄 작업이 프로세스를 자동 시작하는지 확인
3. **중복 방지 테스트**: `run_scheduler.bat`을 수동 실행하여 이미 실행 중일 때 스킵 로그가 기록되는지 확인
