# JooPark Workspace Product Direction

Updated: 2026-06-08

## Direction Hypothesis

JooPark Workspace should be a local-first personal operating cockpit for one operator: today, tasks, notes, projects, local DB knowledge, and LLM reference material should turn into action without a build step, account, or SaaS backend. It should beat broader references by being smaller, faster to ship, easier to self-host, and more honest about local data boundaries.

## References To Beat

| Reference | Strength to learn from | Gap JooPark can exploit |
| --- | --- | --- |
| AFFiNE | Local-first docs, whiteboards, databases, and AI in one knowledge workspace. Source: https://affine.pro/ | Rich but broad. JooPark can stay narrower and put daily execution first instead of becoming a full canvas suite. |
| AppFlowy | Open-source Notion alternative with docs, notes, wikis, databases, Kanban, calendar, and AI. Source: https://docs.appflowy.io/docs/appflowy/readme/welcome-to-appflowy | Full app ecosystem. JooPark can stay no-build/static and keep data ownership visible in every workflow. |
| Linear | Workspace context becomes action through issues, projects, agents, and automations. Source: https://linear.app/docs/linear-agent | Team SaaS orientation. JooPark can make the same execution bias work for one local operator without workspace setup. |

## Loop 1 Decision

The current app already passes the mobile smoke gate, but the home screen order still overexposes launch/evidence machinery before the actual operating cockpit. Loop 1 therefore keeps all verification surfaces intact but moves the daily execution cards ahead of launch operations, tightens the mobile sidebar, and changes quick-start copy toward task/project/backup execution.

## Loop 2 Decision

The next gap was local data safety. Notion keeps deleted pages recoverable from Trash, Linear points accidental issue deletion first to undo and then archives, and AppFlowy has trash events for put-back and restore-all flows. JooPark is a static localStorage app, so Loop 2 adopts the smallest matching safety layer: immediate deletes for events, todos, notes, habits, issues, tasks, queries, and migrations now show an undo toast that restores the deleted record in place. Larger cascading deletes keep their explicit confirmation modal for now.

## Loop 3 Decision

The undo behavior is now a product contract, not a manual smoke note. Loop 3 adds a dedicated release smoke that opens JooPark in an isolated Chrome profile, deletes and restores all immediate-delete record types, verifies task dependency restoration, and confirms restored records are persisted back into localStorage. The release smoke pipeline now runs this gate before accessibility checks.

## Loop 4 Decision

Undo needs a different timeout from ordinary feedback. Loop 4 keeps normal toasts short but gives undo toasts an eight-second window, then strengthens the delete-undo smoke so it waits longer than the old 3.2-second timeout before restoring the first deleted record.

## Loop 5 Decision

Eight-second undo covers the immediate mistake, but it still fails the safer recovery pattern seen in Notion Trash and Linear Recently deleted archives. Loop 5 adds a bounded local recently-deleted ledger for non-cascading deletes and exposes it in Settings, so a single operator can restore or discard deleted events, todos, notes, habits, issues, tasks, queries, and migrations after the undo toast expires.

## Loop 6 Decision

Recovery should not be hidden behind memory of where Settings lives. Notion makes Trash a navigable workspace surface and Linear exposes recently deleted records through archives, while JooPark already has a compact data-safety sheet in the top bar. Loop 6 keeps the action count stable but changes the existing backup/recovery affordance to show the current recently-deleted count and route the operator to Settings when recovery is available.

## Loop 7 Decision

Trash/recently-deleted patterns are not just about restore; they also need a reliable path to discard records. Loop 7 tightens JooPark's local recovery contract so a discarded or cleared ledger entry cannot be restored by a stale undo toast, then extends the release smoke to prove restore, discard, clear, stale-undo blocking, and persistence in one browser run.

## Loop 8 Decision

Recently-deleted recovery should be bounded by time as well as count. Notion and Linear both communicate 30-day recovery windows, so JooPark now keeps at most 40 local recovery records and prunes records older than 30 days during normalization. The Settings panel exposes the 30-day policy and the delete/undo smoke proves expired records are pruned and persisted.

## Loop 9 Decision

Recovery gets slower once the local ledger contains several record types. Notion Trash supports search and filters, and Linear keeps recently deleted records in an archive surface built for quick restore. Loop 9 adds a Settings-level search box and kind filter for recently-deleted records, including filtered empty states, while keeping the ledger local and non-persistent UI filters out of exported workspace data.

## Loop 10 Decision

The recovery surface should be reachable without remembering the Settings route. Notion exposes Trash as a navigable surface and Linear exposes archives through shortcuts, so JooPark now adds a command-palette entry for recently-deleted recovery. The command routes to Settings, focuses the recovery panel after storage-health rerenders, and the delete/undo smoke proves the command path before restore.

## Loop 11 Decision

Local recovery only earns trust if it survives the same backup boundaries as the rest of the workspace. Notion and AppFlowy both treat recovery/data ownership as part of the user's data lifecycle, so JooPark now verifies that `deletedItems` is accepted as a backup slice, appears in import summaries, and trips its 40-record import limit guard. The release smoke covers this without adding another UI surface.

## Loop 12 Decision

Single-item restore is safe but slow when multiple local records were deleted during the same cleanup pass. AppFlowy-style restore-all behavior fits JooPark's local-first recovery ledger if it remains conservative, so Loop 12 adds a Settings "모두 복구" action that restores every restorable ledger entry, leaves failed duplicates behind, and persists both restored records and ledger cleanup. The delete/undo smoke now proves bulk restore before clear/discard paths.

Loop 12 also tightens the route smoke gate so the root development server is not failed by an optional `release-provenance.json` fallback 404. Packaged release provenance remains covered by the release verifier, while local route smoke now stays focused on app routing, layout, console, and real network regressions.

## Loop 13 Decision

Recovery is now tracked as a release-readiness product contract instead of living only in implementation and browser smoke. Loop 13 adds a `recent_deleted_recovery` audit requirement that checks the local ledger, Settings recovery panel, command-palette route, backup/import persistence boundary, and delete/undo smoke proof. The compact audit summary also exposes this contract under Product Contracts, so operator status checks show whether recently-deleted recovery remains pass even when public-launch blockers are still outstanding.

## Loop 14 Decision

The 30-day recovery window should be actionable at item level, not only implied by the Settings panel header. Loop 14 adds per-row expiry metadata and a visible remaining-days badge for recently-deleted records, then extends the delete/undo smoke and release audit contract so every recovery row must expose its expiry timestamp, bounded remaining days, and visible expiry label.

Loop 14 also applies the existing root-dev `release-provenance.json` 404 exception to mobile smoke, not just desktop route smoke. The packaged release still verifies provenance as a release artifact, while local mobile smoke remains focused on responsive layout and interaction regressions.
